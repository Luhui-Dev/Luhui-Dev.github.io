Dino-GSP Gran actualizacion: el dibujo geometrico con IA entra en el flujo de producto

Producto: Pizarra de Geometría Dino-GSP

Version: Open Capabilities 2026-06-29

Fecha de publicacion: 29 de junio de 2026

Ultima actualizacion: 29 de junio de 2026

Dino-GSP publico un conjunto de actualizaciones de capacidades abiertas para desarrolladores el 29 de junio de 2026.

No es solo un boton nuevo ni un endpoint aislado. La actualizacion acerca el dibujo geometrico con IA a un flujo real de producto: los usuarios pueden iniciar el dibujo con IA dentro de tu producto, tu backend puede mantener el control de autenticacion, cuotas y reglas de riesgo, la plataforma abierta puede registrar el uso, el renderizado puede usar una ventana explicita y las figuras generadas por IA pueden seguir una plantilla de estilo compartida.

Para productos EdTech SaaS, bancos de preguntas, productos de matematicas con IA, editores de material didactico y plataformas de contenido curricular, el mensaje practico es claro: el dibujo geometrico con IA ya no tiene que vivir en una herramienta externa. Puede convertirse en un flujo estable dentro de tu propio producto.

Anuncios relacionados:

  • Lanzamiento del dibujo con IA en editor embebido, API y estadisticas de uso
  • Actualizacion de parametros de Render API y plantillas para Agent


Editor embebido: mantener el dibujo con IA dentro del producto

A partir de @dajiaoai/algeo-sdk@2.8.0, el modo de editor embebido puede habilitar el panel de chat con IA.

Eso significa que el editor geometrico dentro de tu producto ya no es solo una pizarra manual. Un usuario puede escribir una solicitud en lenguaje natural dentro del editor, por ejemplo "dibuja el triangulo ABC, construye sus tres medianas y marca el baricentro G", y dejar que la IA genere o modifique la figura geometrica.

Para el usuario final, el flujo es directo: abrir el editor de problemas, describir la figura, verla generada y seguir arrastrando o editando.

Para el equipo de producto, lo importante es que esta capacidad no evita tu sistema. El SDK emite un evento aiRequest. La pagina anfitriona lo escucha y envia la solicitud a tu backend. Tu backend autentica al usuario, valida cuotas y registra logs, y luego llama a la API de dibujo con IA de Dino-GSP para modo embebido. La respuesta en streaming vuelve al editor mediante editor.ai.consumeStream().

La frontera queda clara:

  • El frontend aloja el editor y consume el stream devuelto
  • El backend anfitrion controla identidad de usuario, cuotas de negocio, planes y reglas de riesgo
  • Dino-GSP gestiona llamadas al modelo, generacion de instrucciones geometricas y salida en streaming
  • La plataforma abierta registra uso por aplicacion y consumo de puntos


API y medicion: hacer operable el dibujo con IA

Generar una figura geometrica es solo el primer paso.

Cuando el dibujo con IA entra en un producto real, el equipo necesita respuestas operativas:

  • Que aplicacion hizo las llamadas?
  • Cuanto consumio cada llamada?
  • Como se contabilizan solicitudes canceladas, desconectadas o fallidas?
  • Puede el sistema anfitrion aplicar sus propias cuotas y limites de plan?
  • Se pueden investigar problemas a nivel de aplicacion?

La consola de la plataforma abierta ya soporta uso de IA y consumo de puntos por aplicacion. El flujo de dibujo con IA en editor embebido es puenteado por el SDK, mientras que la facturacion real sigue el uso de tokens de IA producido por el backend de Dino-GSP. Si una solicitud consume tokens de IA, se liquida segun el consumo real.

Esto hace que la integracion sea mas adecuada para operaciones de producto. Un equipo de desarrollo puede usar Dino-GSP como capacidad geometrica de IA subyacente y seguir gestionando cuotas de usuario, planes, limites de llamadas, reglas de riesgo y facturacion de clientes dentro de su propio sistema.

En resumen: Dino-GSP dibuja la figura y aporta evidencia de uso en la plataforma abierta; el producto anfitrion conserva su frontera comercial y de gestion de usuarios.



Render API: usar viewBound para controlar la ventana de exportacion

Cuando una figura geometrica entra en un banco de preguntas, diapositivas, apuntes o reportes, la salida exportada debe ser controlable.

Muchos flujos de diagramas automaticos fallaban en el ultimo paso: la figura estaba generada, pero la imagen exportada tenia demasiado espacio en blanco, quedaba demasiado ajustada, tenia proporciones inconsistentes o requeria recorte manual por problema.

Esta actualizacion mueve las solicitudes de Render API al parametro de ventana viewBound. /api/render, /api/render-svg y /api/render-tikz usan { left, right, bottom, top } para describir la ventana logica. El tamano del lienzo se calcula como width = right - left y height = top - bottom.

Eso importa para la produccion de contenido por lotes.

Un banco de preguntas puede definir limites de exportacion estables para diferentes tipos de problema. Un producto de material didactico puede mantener diagramas con proporcion consistente. Plataformas editoriales y curriculares pueden incorporar las figuras exportadas a una canalizacion automatizada en vez de recortar cada imagen manualmente.

Las respuestas exitosas de renderizado tambien devuelven metadatos mas completos, incluidos objectKey, el viewBound real y scale. Esto facilita archivo, depuracion y reutilizacion.



Plantillas: mantener consistentes los diagramas generados y exportados

Las figuras geometricas en productos educativos deben ser correctas, pero tambien deben verse consistentes.

Si grosores de linea, fuentes, colores, estilos de puntos y etiquetas varian dentro del mismo banco de preguntas, el contenido se siente irregular. El dibujo con IA puede amplificar este problema: una imagen aislada puede verse aceptable, pero la generacion por lotes sin control de estilo compartido crea coste de mantenimiento.

Render API y la API de generacion inteligente ahora soportan el campo template. Render API puede usar template para especificar la plantilla de renderizado, y /api/agent/run tambien acepta un campo opcional template para guiar el estilo de los resultados generados.

Esto es especialmente util para tres tipos de equipos:

  • Equipos de bancos de preguntas y editoriales pueden unificar el estilo geometrico en examenes, soluciones y apuntes
  • Productos EdTech SaaS pueden permitir que diferentes escuelas, tenants o marcas usen estandares visuales distintos
  • Productos de matematicas con IA pueden alinear figuras generadas con la interfaz, plantillas de reporte y activos exportados

Las plantillas convierten el dibujo con IA de "generar una imagen" a algo mas cercano a "generar un activo reutilizable segun las normas del producto".



Donde encaja primero esta actualizacion

Vale la pena evaluar esta actualizacion si tu producto incluye alguno de estos flujos:

  • Editores de bancos de preguntas donde docentes u operadores generan diagramas geometricos directamente desde el texto del problema
  • Productos de resolucion con IA donde los pasos explicativos necesitan geometria visual sincronizada
  • Herramientas de curso y planeacion didactica donde los diagramas se generan en el editor y se exportan a diapositivas o apuntes
  • Plataformas curriculares donde las figuras geometricas deben ser activos editables, exportables y gestionables
  • Productos EdTech SaaS que necesitan geometria profesional sin construir desde cero un modelo geometrico y un motor de pizarra

Antes, los diagramas geometricos solian quedar fuera del flujo de producto como trabajo manual de ilustracion. Ahora pueden entrar en el producto: solicitud del usuario, generacion con IA, presentacion en editor, medicion backend, exportacion con plantilla y reutilizacion como activo de contenido.

Ese es el paso clave para que el dibujo geometrico con IA se convierta en una capacidad real de producto.



Como empezar

Los equipos de desarrollo pueden evaluar la actualizacion con este recorrido:

  • Actualizar a @dajiaoai/algeo-sdk@2.8.0 o posterior
  • Habilitar o mantener el valor predeterminado de ui.aiChatPanel en modo de editor embebido
  • Escuchar el evento aiRequest del SDK y reenviar el payload al backend anfitrion
  • Autenticar al usuario y validar cuota en el backend anfitrion, y luego llamar a la API de dibujo con IA para modo embebido de Dino-GSP
  • Consumir el stream devuelto con editor.ai.consumeStream()
  • Enviar viewBound con el nuevo formato al usar Render API
  • Enviar template cuando los diagramas generados y renderizados necesiten control de estilo compartido
  • Tras el lanzamiento, monitorear uso por aplicacion y consumo de puntos en la consola de la plataforma abierta

Puedes leer la documentacion de SDK, API e integracion con IA en la plataforma abierta de Dino-GSP, o probar primero el dibujo con IA en el sitio principal de Dino-GSP.

Dino-GSP seguira empujando dibujo geometrico, edicion dinamica, generacion con IA, exportacion de formatos e integracion de producto en la misma direccion: convertir los diagramas geometricos de trabajo manual de ilustracion en una capacidad que los productos pueden llamar, gestionar y reutilizar.
