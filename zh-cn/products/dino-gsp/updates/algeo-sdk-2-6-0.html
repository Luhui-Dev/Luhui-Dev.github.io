Algeo SDK 2.6.0 上线：让几何能力更稳地进入你的产品

产品: 大角几何画板

版本: SDK 2.6.0

发布日期: 2026年5月19日

最后修改: 2026年5月19日

我们刚刚发布了 Algeo SDK 2.6.0。

Algeo SDK 是大角几何对外提供的开放能力之一。我们希望它不是一个孤立的编辑器，而是一套可以被教育产品、AI 数学产品、题库系统和教研工具接进去的几何能力：能画、能展示、能交互，也能进入你自己的内容和业务流程。

过去一段时间，我们和一些教育产品、AI 数学产品、题库系统的团队交流时，听到的反馈很朴素。

大家并不是只问 SDK 有哪些 API。更多时候，他们问的是：

  • 老师画好的图，能不能直接放到课件、讲义和试卷里？
  • 学生打开题目时，画板会不会偶尔卡住？
  • AI 生成的几何步骤，能不能导出成解析图或报告图？
  • 同一份几何素材，能不能既能交互，又能沉淀为平台里的内容资产？
  • 接入过程中遇到异常，SDK 能不能自己处理得更干净一点？

这些问题听起来都不复杂，但对一个要上线的产品来说，每一个都很重要。

所以这个版本，我们主要做了两件事：增加画板图片导出能力，以及提升嵌入初始化的稳定性。同时，我们也把示例和文档补了一轮，希望开发团队在本地调试、写 demo、做上线前验证时，能更顺一些。

如果你想直接试，可以使用 @dajiaoai/algeo-sdk@2.6.0。图片导出的演示案例也已经放出来了：Export Slide Image Demo。

如果你们的产品里有几何题、几何课件、AI 解题、题库编辑、作业报告或教研内容生产，下面这些场景应该会比较熟悉。



场景一：老师做完动态几何，也能一键进入课件和讲义

很多教育产品都希望给老师提供交互式几何画板。

但老师真正的工作流并不只停在画板里。她可能要把图放进 PPT，放进讲义，放进试卷，放进课后作业解析，甚至放进公众号或校本资源库。

过去，动态图和静态内容之间经常隔着一道墙：画板里能动，但导出时又要截图；截图能用，但清晰度、尺寸、批量处理和复用都不稳定。

Algeo SDK 2.6.0 新增了画板图片导出能力，让产品团队可以把几何画板里的内容导出为图片，并进一步流转到自己的课件、讲义、试卷、报告和内容管理系统中。

这意味着老师在你的产品里完成的几何素材，不再只是一个"在线画板状态"，也可以变成一份可发布、可归档、可传播的教学资产。

对教育产品团队来说，这个能力非常实际：

  • 课件工具可以把几何图直接插入幻灯片
  • 题库系统可以生成标准题图和解析图
  • 教研平台可以把动态素材沉淀为静态预览
  • 作业系统可以在报告里展示关键图形
  • 内容团队可以把同一份几何素材用于多端发布

几何内容从此不只是"能交互"，也更容易进入真实的内容生产链路。

我们也准备了一个可用的导出图片演示，方便你先看效果再评估流程：12-export-slide-image.html。



场景二：题库和作业系统，需要更稳定的嵌入体验

题库、作业和测评系统对稳定性的要求很高。

学生打开一道题时，几何图不能偶尔加载失败；老师批改作业时，画板不能卡在初始化状态；运营人员配置内容时，也不能靠刷新页面来赌一次成功。

Algeo SDK 2.6.0 对嵌入初始化流程做了稳定性提升，尤其是针对 iframe 加载、超时处理、异常清理等接入过程中常见的问题。

这些变化对最终用户来说可能不显眼，但对产品团队很重要。

因为在真实产品里，一个几何组件不是只在演示页面打开一次，而是会出现在大量页面、不同网络、不同设备、不同权限状态和复杂的单页应用路由中。

更稳定的嵌入能力，意味着：

  • 学生端打开题目时，几何内容加载更可预期
  • 老师端编辑题目时，异常状态更少打断工作
  • 平台方排查问题时，链路更清晰
  • 产品可以更放心地把几何组件放到核心流程里

对教育平台来说，稳定不是锦上添花，而是几何能力能不能进入主流程的前提。



场景三：AI 数学产品，不只需要解出来，还要呈现出来

AI 数学产品正在从"给答案"走向"给过程"。

尤其是几何题，学生真正需要的往往不是一句"连接某点并作辅助线"，而是看到这条线为什么要画、画完之后关系如何变化、不同构型之间有什么联系。

当 AI 能生成几何过程，下一步就会遇到内容呈现的问题：

  • 生成的图能不能稳定嵌入在解题页？
  • 多步讲解能不能拆成不同画板或步骤？
  • 每一步能不能导出为图片，用在解析、报告或分享页里？
  • 学生和老师能不能在同一个几何内容上继续编辑、复盘和二次创作？

Algeo SDK 2.6.0 对"嵌入可靠性"和"图片导出"的增强，正好补上了这条链路。

它让 AI 生成的几何内容，不只停留在一次性的页面展示里，而是可以进入更完整的产品闭环：生成、展示、交互、导出、沉淀、复用。

对 AI 数学产品来说，这会直接影响用户感知。

因为用户不会关心背后调用了什么模型，也不会关心画板是怎么初始化的。他们只会感受到：这道几何题讲得清不清楚，图能不能打开，步骤能不能看懂，内容能不能带走。



场景四：教研和内容团队，需要把几何素材变成可管理资产

在很多学校、机构和内容公司里，几何素材长期是"散落的文件"。

一部分在老师电脑里，一部分在 PPT 里，一部分是截图，一部分是旧软件源文件，还有一部分在题库系统里但无法复用。

这会带来一个很现实的问题：几何内容很难资产化。

Algeo SDK 的价值，不只是把一个画板嵌进产品，而是把大角几何已经积累下来的动态几何能力开放出来，让几何素材可以纳入你自己的业务系统：

  • 谁创建了这份素材
  • 属于哪个年级、章节、知识点
  • 哪些老师可以编辑
  • 哪些内容可以发布给学生
  • 哪些图可以导出到讲义和试卷
  • 哪些素材可以进入 AI 生成或推荐流程

2.6.0 对导出和嵌入链路的增强，会让这套资产化过程更顺。

内容团队可以在自己的 CMS 里创建几何图，导出标准图片用于出版和分发；教研平台可以为每份动态几何素材生成预览图；题库系统可以让同一份内容同时服务编辑端、学生端和打印端。

几何不再是产品之外的"配图工作"，而是可以进入平台数据结构和内容流程的一等资产。



场景五：产品团队真正想要的，是少操心

对一个准备上线的产品团队来说，选择一个 SDK，表面上是在选择能力，实际上是在选择一段长期合作关系。

你当然会关心它能不能画点线圆、能不能编辑、能不能展示、能不能和业务系统通信。

但上线之后，你更关心的是：

  • 用户打开时是否稳定
  • 出问题时是否能被正确处理
  • 示例和文档是否跟得上
  • 新同事接手时是否容易理解
  • 未来版本是否会持续补齐真实场景里的缺口

Algeo SDK 2.6.0 同步更新了示例、配置支持和文档，让接入体验更贴近真实项目，而不是只服务一个最简单的 demo。

这件事很小，也很关键。

因为 SDK 不是给技术同学"看一眼"的，它最终要进入一个产品团队的协作流程：产品经理要确认场景，前端要接入组件，后端要处理数据归属，运营要配置内容，客户成功要解释效果。

文档和示例越接近真实流程，产品团队走到上线的路就越短。



Algeo SDK 2.6.0 适合哪些团队关注

如果你的产品里已经出现以下任意场景，就值得了解 Algeo SDK：

  • 在线教育平台：需要在课程、题库、作业或课堂工具中加入动态几何能力
  • AI 数学产品：希望把几何题从文字解析升级为可视化、可交互的解题过程
  • 题库和测评系统：希望几何题不再只是静态图片，而是可编辑、可展示、可沉淀的数据
  • 教研协作平台：希望老师共建、复用和管理几何教学素材
  • 教辅出版和内容生产团队：希望提高几何配图、课件图、讲义图和试卷图的生产效率
  • 教育 SaaS 厂商：希望低成本补齐专业几何能力，而不是从零开发一套画板

一句话：只要你的产品需要处理"几何内容"，Algeo SDK 都可以把大角几何的开放能力接入到你的底层流程里。



写在最后

Algeo SDK 2.0 让几何画板可以嵌进产品里。

Algeo SDK 2.6.0 则让这件事更进一步：嵌入更稳定，内容更容易导出，示例和文档更贴近产品团队的真实工作。

这不是一次面向技术参数的更新，而是一次面向产品落地的更新。

因为教育产品真正需要的不是一个孤立工具，而是一种可以进入业务流程的几何能力：能被老师使用，能被学生打开，能被内容团队复用，能被 AI 生成，能被平台管理，能在关键时刻稳定工作。

Algeo SDK 2.6.0 已正式上线。

如果你正在为自己的教育产品补齐几何能力，现在是一个很适合开始接入的版本。接入时可以直接使用 @dajiaoai/algeo-sdk@2.6.0。
