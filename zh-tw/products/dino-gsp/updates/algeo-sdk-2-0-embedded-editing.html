大角幾何重磅更新：Algeo SDK 2.0 內嵌編輯模式正式上線

產品: 大角幾何畫板

版本: SDK 2.0

發布日期: 2026年5月8日

最後修改: 2026年5月10日

影片可以嵌入，文件可以嵌入，表格可以嵌入。

但幾何呢？

過去十年，只要你的產品需要讓使用者畫一道幾何題、改一個動態圖形、儲存一份帶互動的幾何素材，結局基本只有一個：跳出去用別的工具，做完截圖，再貼回來。這條斷裂的工作流，橫在了所有教育、教研、AI 數學產品面前。

今天，Algeo SDK 2.0 內嵌編輯模式 正式上線。從這一天起，幾何不再是嵌入的盲區。它可以像一個標準元件一樣，被裝進你的產品裡，資料回流到你的業務，UI 融入你的設計，權限走你自己的體系。

下面是我們看到的五種典型場景。如果其中任何一個讓你想起自己的產品，這篇文章就值得你看完。



場景一：線上教育平台，讓老師在你的產品裡直接出題

一位高中數學老師在你們平台準備明天的幾何課。她需要一道「圓中的角度證明」作為例題。

過去：她打開幾何畫板軟體，做完圖，截圖，貼回你們的題庫後台。文字描述貼一邊，圖片貼另一邊。學生看到的是一張靜態圖，拖不動、改不了、考完試就沒用了。

現在：她在你們題庫後台點「插入幾何畫板」，右側 Algeo 編輯器原地展開。圓、點、輔助線一氣呵成，點儲存，畫板資料進入你們的題庫，綁定到她的帳號、所屬學校、教材章節。

學生端打開這道題時，可以直接拖動圓上的點，親眼看到角度怎麼變化。整個過程，你的產品始終在場：資料是你們的，權限是你們的，內容版權是你們的，使用者行為日誌也是你們的。



場景二：AI 數學產品，讓 AI 和學生在畫板上對話

這是過去一年裡我們看到成長最快的一類客戶。

學生拍了一道幾何題上傳，你們的 AI 解析出題意，生成解題思路。但只有文字解釋是不夠的，學生需要看到輔助線為什麼這麼畫，也需要親手驗證當某個點動起來時，等量關係是不是真的成立。

Algeo 編輯模式讓這個迴路第一次能夠閉合：

  • AI 解析後，程式碼自動生成畫板內容並載入編輯器
  • 學生在你們的產品裡直接互動，拖動、修改、嘗試
  • 學生的每一次改動都透過事件回傳給你們，可以再投餵給下一輪 AI 分析
  • AI 給出針對學生具體改動的回饋，而不是泛泛而談

教育的本質是回饋循環。文字加靜態圖的回饋循環已經做不下去了，幾何這塊拼圖缺的就是一個能被程式碼驅動、又能讓學生動手的畫板。



場景三：教輔出版與內容生產，幾何素材生產納入正軌

一家教輔出版公司過去的內容生產流程裡，幾何插圖是個獨立小作坊：作者畫圖，美編重做向量，編輯校對，版式設計師再處理一次。每一道題的幾何素材要在四個工具、五個人手裡轉一圈。

把 Algeo 編輯器嵌進他們的內容管理系統後，整條流水線被壓扁：

  • 作者直接在 CMS 裡寫題、畫圖，所有素材是結構化的幾何資料，不是圖片
  • 編輯校對時可以直接調出原畫板修改，不用回頭找作者
  • 同一份幾何資料可以一次性匯出到 PDF、Web、印刷、互動課件，畫一次，處處可用
  • 版本控制走 CMS 自己的那套，幾何畫板不再是體外資產

對內容機構來說，這不是省一個工具的問題，而是把幾何資產化的問題。



場景四：校內與機構教研，集體備課從此有了幾何素材

教研活動有一個很老的痛點：語文備課組有自己的素材庫，英語有自己的語料庫，數學有題庫，但幾何始終是個孤兒。每位老師電腦裡散落著幾十個幾何畫板源文件，離職就帶走，換電腦就丟，新老師無從繼承。

機構把 Algeo 編輯器嵌入自己的教研協作平台後：

  • 幾何素材進了機構的資產庫，按學科、年級、知識點歸檔
  • 老師之間可以基於同一份畫板二次創作，留下完整的修改歷史
  • 新老師入職第一天就能拿到資深老師沉澱下來的幾何資源
  • 走自己的權限和審批流，哪些可以共享、哪些只對學科組開放，完全你說了算



場景五：題庫與作業系統，幾何題成為一等公民

很多題庫系統裡，選擇題、填空題、解答題都有結構化模板，唯獨幾何題是張圖。這意味著：

  • 幾何題沒法做相似題推薦，系統不知道兩道題在數學結構上是不是同一類
  • 幾何題沒法做精細批改，學生交上來的也是張圖
  • 幾何題沒法做學情分析，你不知道學生是在哪一步卡住的

Algeo 把幾何題變成結構化資料後，這些事第一次成為可能：

  • 題目和答題過程都是結構化的，題庫能像處理代數題那樣處理幾何題
  • 學生作答時的每一次操作都可以回傳，批改系統能定位到「學生在第幾步動了哪個點」
  • 學情分析能告訴老師：這個班級 70% 的學生沒想到要做這條輔助線



技術上，我們準備好了什麼

場景聽起來誘人，但落地永遠是工程問題。Algeo SDK 2.0 在以下幾個維度做到了生產級別。

雙向資料通訊，資料歸屬清晰

畫板裡的每一次編輯、畫板切換、儲存請求都透過 postMessage 即時回傳到宿主。儲存按鈕的控制權交給你，iframe 不會越過你的業務直接持久化任何東西。什麼時候儲存、存到哪裡、需要什麼權限，都由你的服務端決定。SDK 只維護「已儲存 / 未儲存」這個 UI 狀態。

UI 完全可自訂，融入你的產品

導覽列、畫板列表、工具箱、代數面板、文件面板五個區域獨立開關，執行時可改。你可以讓編輯器在 AI 輔助場景下「隱身」成一塊純畫布，也可以在專業出題場景下展開完整工具鏈。極端情況下，你甚至可以用自己的 UI 替換我們的畫板列表，只用 SDK 提供的能力 API 來驅動。

工程化的能力分層

SDK 把編輯器能力拆成了四組清晰的 unit：畫板文件 document、多畫板管理 slides、歷史記錄 history、展示模式 mode。每一組能力都可以被獨立呼叫，也意味著未來每一組都可以獨立增強，而不會破壞其他部分的契約。

協議版本化，長期可演進

每一次 SDK 與 iframe 的握手都會帶上協議版本號。這意味著你今天的整合，在我們升級三次之後仍然能跑，也意味著我們能持續為你交付新能力，而不需要你每次都重寫程式碼。

面向生產的健壯性

30 秒初始化逾時保護、標準化錯誤碼、乾淨的銷毀生命週期、自部署宿主支援 baseUrl。這些細節決定了 SDK 能不能扛住真實世界的網路抖動、CSP 策略、單頁應用的複雜路由切換。我們已經在多個客戶的生產環境裡跑過驗證。



為什麼選大角幾何

國內能做出動態幾何這個層級編輯器的團隊屈指可數。我們花了一整年時間把它打磨到生產可用，又花了一整個版本週期把它從產品變成元件。只有當它能被你裝進任何產品裡，幾何這個領域才真正打開。

如果你正在做的產品裡有「幾何」這個詞，不管是 K12、高等教育、AI 數學、教輔出版還是教研工具，歡迎來聊聊。
